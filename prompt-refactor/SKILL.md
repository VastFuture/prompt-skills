# Prompt Refactor

把生产级 prompt 当成一段会执行的系统代码来重构，而不是当成说明文去润色。

**核心原则：** 先看全稿，再动局部。先收骨架，再改句子。少描述，多定义。少补丁，多主轴。

## When to Use

在这些场景使用：

- prompt 越写越长，规则越来越碎，模型开始不稳定
- 同一类约束散落在多个章节，互相重复或互相打架
- 输出有明显 AI 模板腔、客服机器人腔、机械模仿用户问题
- 模型把内部判断过程、内部分类、内部路由说给用户听
- SOP 写成了“给人看的讲义”，而不是“给模型执行的规则”
- 每修一个 case 就补一条规则，prompt 变成 bug 目录

不要在这些场景使用：

- 只改一个局部业务规则，且全稿结构本身稳定
- 只是补一个 few-shot，不涉及结构收敛
- 单纯做文案润色，不涉及执行稳定性

## Workflow

### 1. 先通读全稿

不要盯着一两句坏话术直接修。

先识别这份 prompt 里混了哪些层：

- 执行规则
- 给维护者看的解释
- 对客示例
- 边界映射
- 禁止事项

如果这些层混在一起，局部修词只会继续打补丁。

### 2. 标记系统性问题

优先找这几类问题：

- **重复**：同一类规则在多个章节重复出现
- **漂移**：前面要求 A，后面又补了 B，模型抓不住主轴
- **解释层过厚**：大量“为什么这样做”的说明文
- **边界泄漏**：内部判断轴容易被复述给用户
- **模板腔**：固定句式、虚假客套、客服套词过多

### 3. 先收骨架，再改文案

重构顺序固定：

1. 合并重复模块
2. 删除解释层
3. 收拢统一出口
4. 统一 SOP 模板
5. 最后才改示例和具体句子

不要反过来做。

## Refactor Heuristics

### A. 只保留一处“对客表达出口”

对客表达规则、边界外回复规则、推荐表达、避免表达，应尽量集中在一个模块。

不要把它们散在：

- 总体原则
- 场景 SOP
- 边界控制
- 禁止事项

### B. 边界规则分成“内部决策”和“外部出口”

内部决策写：

- 何时命中
- 何时切流
- 何时查工具

外部出口只写：

- 给用户怎么说

**禁止**把内部原因写成用户可复述的话。

### C. SOP 统一成同一种模板

推荐统一成：

- 适用场景
- 生效前提
- 处理链条
- 表达要求

如果有的场景写成散文，有的写成流程图，有的写成讲解文，模型很容易漂。

### D. 少描述，多定义

坏写法：

- 解释为什么要这样做
- 反复讲这一步的目的
- 给模型讲方法论

好写法：

- 触发条件
- 执行动作
- 结果分支
- 禁止事项

### E. 示例要少而强

示例不要多。

保留两类就够：

- **推荐表达**：短、自然、像真人
- **避免表达**：暴露内部规则、模板腔、模仿式表达

与其给十条一般示例，不如给三条强示例。

## Template Smell Checklist

看到这些，优先怀疑有“AI 腔”：

- `这边`
- `建议您`
- `进一步`
- `久等了`
- `关于您提到的`
- `您想了解……的需求`
- 固定 `理解 + 建议` 句式
- 过度解释用户当前所处阶段

注意：

- 不是所有词都要机械禁掉
- 要看它是否在当前 prompt 里形成了“稳定模板腔”

## Rewrite Rules

改 prompt 时优先遵守这些规则：

1. 一类规则只放一处
2. 用户看不到的逻辑，不要写成用户会复述的句子
3. 先删解释，再收规则
4. 先保证执行稳定，再追求语言漂亮
5. 任何“为了修某个 case 再加一条”的冲动，都先问：能不能抽成更高层约束

## Common Mistakes

### 1. 只改坏句子，不改结构

结果：prompt 越修越碎。

### 2. 把方法论写进执行 prompt

结果：模型读到一堆“为什么”，却抓不到“怎么做”。

### 3. 在多个章节重复同一条约束

结果：注意力漂移，权重分散。

### 4. 用很多“温柔话术”解决模板腔

结果：更像 AI 客服，而不是更像真人客服。

## Output Standard

做完重构后，至少确认：

- 对客表达是否只有一个统一出口
- 边界控制是否只保留内部映射
- SOP 是否用同一种模板
- 是否删掉了重复解释层
- 是否还存在明显模板腔或内部规则泄漏

如果一个 prompt 改完以后，主干结构仍然需要“靠解释理解”，说明还没重构完。
