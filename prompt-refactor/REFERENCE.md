# Prompt Refactor Reference

轻量参考，不是长讲义。只保留最常复用的重构模板。

---

## Before / After

### Bad: 规则分散 + 解释过厚 + 话术模板腔

```markdown
## 对客表达约束
- 不要暴露内部分类

## 总体原则
- 边界外场景只说用户该做什么

## 边界控制
- 用车中问题不属于售后
- 原因：问题仍发生在用车过程中...

## 推荐表达
- 理解您想了解……的需求，这类情况建议您尽快联系人工客服协助处理。
```

问题：

- 同一类约束散在多处
- “原因”是写给人看的，不是写给模型执行的
- 推荐表达会把模型带成客服机器人腔

### Better: 单一出口 + 内外分层 + 少量强示例

```markdown
## 对客表达约束
- 只对用户说结论、依据和下一步
- 边界外场景只说下一步动作，不解释内部原因
- 避免模板腔和模仿式表达

**边界外推荐表达：**
- 这个问题需要人工客服继续帮您核实。
- 麻烦您联系人工客服继续跟进。

## 边界控制
1. 用车中紧急问题 -> EMERGENCY
2. 未取车前退款 -> ORDER_MANAGEMENT
```

特点：

- 对客出口只有一处
- 内部决策和外部表达分开
- 规则少，但权重更集中

## Refactor Template

重构生产 prompt 时，优先收成这个骨架：

```markdown
## 0. 对客表达约束
- 对外只说什么
- 不能说什么
- 推荐表达
- 避免表达

## 1. 角色定义

## 2. 核心职责范围

## 3. 上下文 / 工具

## 4. 总体处理原则

## 5. 场景 SOP
### 场景 A
- 适用场景
- 生效前提
- 处理链条
- 表达要求

### 场景 B
- 适用场景
- 生效前提
- 处理链条
- 表达要求

## 6. 边界控制
- 只保留内部决策映射

## 7. 禁止事项
```

## Quick Checks

改完后，至少自查这 8 项：

1. 对客表达是否只有一个统一出口
2. 边界控制是否只保留内部映射
3. 是否删掉了“为什么这样做”的解释层
4. SOP 是否都用了同一种模板
5. 是否还存在明显模板腔
6. 是否还有内部判断轴可能被复述给用户
7. 是否还在为单个 case 继续补丁
8. 是否比改前更短、更清楚、更可执行

---

## High-Risk Smells

看到这些，优先怀疑 prompt 会把模型带歪：

- 同一条规则在 3 个以上章节重复
- “推荐表达”比“执行规则”还长
- 边界控制里大量写“原因”
- 句子里反复出现：
  - `这边`
  - `建议您`
  - `进一步`
  - `久等了`
  - `您想了解……的需求`
  - `关于您提到的……问题`

---

## Rewrite Order

永远按这个顺序改：

1. 合并重复模块
2. 删除解释层
3. 收拢统一出口
4. 统一 SOP 模板
5. 最后改具体句子

不要反过来。
