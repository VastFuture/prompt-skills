# Prompt Refactor Examples

只放典型案例，帮助快速判断“哪里该收、怎么收”。

---

## Case 1: 客服分流 Prompt

**原始问题**

- 对客表达规则分散在多个章节
- 边界控制里写了很多“原因”
- 推荐话术明显带客服机器人腔

**坏味道**

```markdown
## 总体原则
- 边界外场景只说用户该做什么

## 边界控制
- 用车中问题不属于售后
- 原因：问题仍发生在用车过程中...

## 推荐表达
- 理解您想了解……的需求，这类情况建议您尽快联系人工客服协助处理。
```

**重构动作**

1. 把对客表达规则、推荐表达、避免表达并到一个模块
2. 边界控制只保留内部映射，不写“原因”
3. 把话术压成短句，去掉 `这边`、`建议您`、`久等了`

**更优结果**

```markdown
## 对客表达约束
- 边界外场景只说下一步动作
- 不解释内部判断轴

**边界外推荐表达：**
- 这个问题需要人工客服继续帮您核实。
- 麻烦您联系人工客服继续跟进。

## 边界控制
1. 用车中紧急问题 -> EMERGENCY
2. 未取车前退款 -> ORDER_MANAGEMENT
```

## Case 2: 工具型 Agent Prompt

**原始问题**

- prompt 一边讲工具调用规则，一边讲工具设计思路
- “为什么先查 A 再查 B”写了很多段解释
- 真正的执行顺序被淹没

**坏味道**

```markdown
先调用 search，因为这样可以减少误判。
之所以不先调用 fetch，是因为 search 可以帮助模型建立更全面的理解。
如果 search 的结果不够好，再考虑 fetch。
```

**重构动作**

1. 删除“为什么这样做”的长解释
2. 改成固定执行顺序
3. 只保留触发条件和分支结果

**更优结果**

```markdown
## 工具顺序
1. 先调 `search`
2. 结果不足时再调 `fetch`
3. 若仍无依据，明确说明暂未查到
```

## Case 3: 知识查询型 Prompt

**原始问题**

- 同时在讲“如何提问”“为什么不能写关键词”“查询结果怎么处理”
- 规则散在多个小节
- 模型容易输出超长 query 或自己拼知识

**坏味道**

```markdown
在查询时，你需要先理解用户真正想问的问题。
因为很多人会写关键词，而关键词会导致搜索质量下降。
所以你要尽量写完整的问题句子。
```

**重构动作**

1. 把查询规则收敛成 5 到 7 条硬规则
2. 删掉“因为……所以……”的解释句
3. 用好/坏 query 对照替代大段说教

**更优结果**

```markdown
## 查询规则
1. 先确定查询焦点
2. 默认用 1 条短、完整、聚焦的自然语言查询
3. 同时涉及多个焦点时，拆成 2 到 3 条短查询
4. 禁止只写关键词串
5. 结果不明确时，明确说明暂未查到，不要猜测
```

## Reuse Pattern

如果你看到一份 prompt 同时满足下面 3 条，优先用本 skill：

1. 规则分散
2. 解释过厚
3. 局部修补越来越多

这类 prompt 最需要的不是“再补一句”，而是先做结构重构。
