# prompt-refactor — 提示词重构

> **一句话描述：** 对生产级提示词进行系统性重构。

---

## 核心理念

> "把 prompt 当成一段会执行的系统代码来重构，而不是当成说明文去润色。"

**核心原则：** 先看全稿，再动局部。先收骨架，再改句子。少描述，多定义。少补丁，多主轴。

---

## 5 个预警信号

当你看到这些信号时，**必须**用这个技能重构：

### ⚠️ 信号 1：prompt 越写越长，规则越来越碎

> 每次修一个 case 就补一条规则，prompt 变成了 bug 目录。

### ⚠️ 信号 2：同一类约束散落多处互相打架

> 对客表达规则分散在多个章节，前面要求 A，后面又补了 B。

### ⚠️ 信号 3：输出有 AI 模板腔/客服机器人腔

> 固定句式、虚假客套、过度解释用户当前所处阶段。
>
> 典型模板腔：
> - `这边`、`建议您`、`进一步`
> - `久等了`、`关于您提到的`
> - `您想了解……的需求`

### ⚠️ 信号 4：模型泄漏内部判断逻辑给用户

> 模型把内部分类标准、内部分流规则说给用户听。

### ⚠️ 信号 5：SOP 写成了"给人看的讲义"

> 而不是"给模型执行的规则"。

---

## 重构 5 步法

```
① 合并重复模块 → ② 删除解释层 → ③ 收拢统一出口 → ④ 统一SOP模板 → ⑤ 改具体句子
```

### ① 合并重复模块

同一类规则只放一处。

**常见需要合并的内容：**
- 对客表达规则
- 边界外回复规则
- 推荐表达
- 避免表达

### ② 删除解释层

删掉所有"为什么这样做"的说明文。

**保留：**
- 触发条件
- 执行动作
- 结果分支
- 禁止事项

**删除：**
- 解释为什么要这样做
- 反复讲这一步的目的
- 给模型讲方法论

### ③ 收拢统一出口

**对客表达只有一个统一出口**，不要分散在：
- 总体原则
- 场景 SOP
- 边界控制
- 禁止事项

### ④ 统一 SOP 模板

所有场景 SOP 用同一种模板：

```
### 场景名称
- 适用场景
- 生效前提
- 处理链条
- 表达要求
```

如果有的场景写成散文，有的写成流程图，有的写成讲解文，模型很容易漂。

### ⑤ 改具体句子

最后才改示例和具体句子。

> 与其给十条一般示例，不如给三条强示例。

---

## 4 条改写铁律

### 1. 一类规则只放一处

同一约束在多个章节重复 → 注意力漂移，权重分散。

### 2. 用户看不到的逻辑，不写成用户会复述的句子

**内部决策**写：
- 何时命中
- 何时切流
- 何时查工具

**外部出口**只写：
- 给用户怎么说

**禁止**把内部原因写成用户可复述的话。

### 3. 先删解释，再收规则

改 prompt 时优先：
1. 删除"为什么"的解释
2. 收拢同类规则
3. 最后才改具体措辞

### 4. 先保证执行稳定，再追求语言漂亮

不要用很多"温柔话术"解决模板腔，结果更像 AI 客服。

---

## 适用场景

- prompt 越写越长，规则越来越碎，模型开始不稳定
- 同一类约束散落在多个章节，互相重复或互相打架
- 输出有明显 AI 模板腔、客服机器人腔、机械模仿用户问题
- 模型把内部判断过程、内部分类、内部路由说给用户听
- SOP 写成了"给人看的讲义"，而不是"给模型执行的规则"
- 每修一个 case 就补一条规则，prompt 变成 bug 目录

## 不适用场景

- 只改一个局部业务规则，且全稿结构本身稳定
- 只是补一个 few-shot，不涉及结构收敛
- 单纯做文案润色，不涉及执行稳定性

## Before / After 对比

### 改写前：规则分散 + 解释过厚 + 模板腔

```markdown
## 总体原则
- 边界外场景只说用户该做什么

## 边界控制
- 用车中问题不属于售后
- 原因：问题仍发生在用车过程中...

## 推荐表达
- 理解您想了解……的需求，这类情况建议您尽快联系人工客服协助处理。
```

### 改写后：单一出口 + 内外分层 + 少量强示例

```markdown
## 对客表达约束
- 只对用户说结论、依据和下一步
- 边界外场景只说下一步动作，不解释内部原因
- 避免模板腔和模仿式表达

**边界外推荐表达：**
- 这个问题需要人工客服继续帮您核实
- 麻烦您联系人工客服继续跟进

## 边界控制
1. 用车中紧急问题 -> EMERGENCY
2. 未取车前退款 -> ORDER_MANAGEMENT
```

## 重构骨架模板

重构生产 prompt 时，优先收成这个骨架：

```markdown
## 0. 对客表达约束
- 对外只说什么
- 不能说什么
- 推荐表达
- 避免表达

## 1. 角色定义

## 2. 核心职责范围

## 3. 上下文 / 工具

## 4. 总体处理原则

## 5. 场景 SOP
### 场景 A
- 适用场景
- 生效前提
- 处理链条
- 表达要求

## 6. 边界控制
- 只保留内部决策映射

## 7. 禁止事项
```

## 改完后自检清单

至少自查这 8 项：

- [ ] 对客表达是否只有一个统一出口
- [ ] 边界控制是否只保留内部映射
- [ ] 是否删掉了"为什么这样做"的解释层
- [ ] SOP 是否都用了同一种模板
- [ ] 是否还存在明显模板腔
- [ ] 是否还有内部判断轴可能被复述给用户
- [ ] 是否还在为单个 case 继续补丁
- [ ] 是否比改前更短、更清楚、更可执行

---

## 常见误区

| 误区 | 后果 |
|------|------|
| 只改坏句子，不改结构 | prompt 越修越碎 |
| 把方法论写进执行 prompt | 模型读到一堆"为什么"，却抓不到"怎么做" |
| 在多个章节重复同一条约束 | 注意力漂移，权重分散 |
| 用很多"温柔话术"解决模板腔 | 更像 AI 客服，而不是更像真人 |
