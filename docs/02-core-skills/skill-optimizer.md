# skill-optimize — 技能优化器

> **一句话描述：** 优化现有 skill 的结构与设计。

---

## 核心理念

> "先审查，再规划，最后在确认后修改。"

**设计模式：**
- **Reviewer**：先判断目标 skill 的问题，再给诊断
- **Inversion**：先出计划并等待确认，不直接改文件
- **Generator（轻度）**：在用户确认后，输出更清晰的 skill 结构或工作流

---

## 5 步标准流程

```
① 确定范围 → ② 审查 → ③ 输出计划 → ④ 确认后实施 → ⑤ 校验
```

### ① 确定范围（Scope）

先确认目标 skill 和本次优化范围：

- 优先采用用户明确提出的优化方向（触发词、description、结构拆分、脚本化、确认流程）
- 如果用户只说"优化这个 skill"，先做完整审查，再给出分优先级计划
- 如果目标 skill 不明确，只问一个最短问题

### ② 审查（Review）

读取目标 skill 的 `SKILL.md`，按需读取关联的 `references/`、`scripts/`、`assets/`。

**审查基线：**
- `references/review-checklist.md` — 审查清单
- `references/skill-design-review-framework.md` — 设计模式判断
- `references/技能创作最佳实践 - Claude API Docs.md` — 最佳实践取舍

**重点检查：**
- `name` 是否为小写连字符格式，且与目录一致
- `description` 是否同时说明"做什么"和"何时用"
- 当前 skill 属于哪种主模式
- 工作流是否清晰、可执行
- 内容是否过胖、重复
- 输出格式是否明确

### ③ 输出计划（Plan）

先给诊断，再给计划，不要直接改文件。

**必须包含两部分：**

```
# Skill 审查结论

## 模式判断
- 主模式：...
- 次模式：...
- 当前判断：模式匹配 / 模式错位 / 模式不清

## 高优先级
- [问题] 影响触发、正确性或执行稳定性

## 中优先级
- [问题] 影响可维护性、可复用性或上下文成本

## 低优先级
- [问题] 体验提升项，可选

# 优化计划
1. 修改 [目标文件路径]
   - 变更内容：
   - 原因：
   - 是否受用户指定方向驱动：是/否

请确认是否按以上计划执行。
```

**规则：**
- 未获得确认前，不要修改目标 skill
- 如果发现超出本次范围的大问题，单独列为"额外建议"

### ④ 确认后实施（Implement）

在用户确认后，再修改目标 skill：

- 优先改动计划中已确认的文件
- 尽量小步重构；仅在确有收益时拆分新的文件
- 保留用户原有有效内容
- 只删除重复、失效或与新流程冲突的部分

### ⑤ 校验（Verify）

改完后完成校验：

- frontmatter 只包含 `name` 和 `description`
- `name`、目录名、触发语义一致
- `description` 能独立表达触发条件
- `SKILL.md` 主体比改动前更短、更清晰
- 如有"先审查后确认再修改"的门槛，是否在说明里写清楚

## 5 种 Skill 模式判断

每个 skill 都有主模式和次模式，判断模式是否匹配任务本身。

| 模式 | 说明 | 典型场景 |
|------|------|----------|
| **Tool Wrapper** | 封装外部工具调用 | `search`、`code-runner` |
| **Generator** | 从零生成内容 | `prompt-craft` |
| **Reviewer** | 判断问题，给出诊断 | `code-reviewer` |
| **Inversion** | 反向引导，追问澄清 | `planner`、`ask` |
| **Pipeline** | 串联多个步骤 | `autopilot`、`team` |

**模式错位风险：**
- 用 Generator 模式做 Reviewer 任务 → 缺少诊断，直接给方案
- 用 Reviewer 模式做 Inversion 任务 → 只分析不行动

---

## 审查清单概览

### 维度 1：命名规范
- `name` 是否为小写连字符格式
- 是否与目录名一致
- 触发语义是否一致

### 维度 2：触发描述
- `description` 是否同时说明"做什么"和"何时用"
- 是否便于触发

### 维度 3：内容组织
- 是否过胖、重复
- 是否适合拆到 `references/`、`scripts/`、`assets/`
- 是否缺少必要参考资料

### 维度 4：工作流
- 是否清晰、可执行
- 是否存在必须确认却没卡住的步骤
- 是否缺少 Gotchas、checklist、模板

### 维度 5：模式匹配
- 主模式是否匹配任务本身
- 次模式是什么
- 模式是否清晰

### 维度 6：输出质量
- 输出格式是否明确
- 是否方便另一个 agent 稳定执行
- 是否存在过度设计、railroading

---

## 审查优先级

快速判定时的处理顺序：

| 优先级 | 问题类型 | 示例 |
|--------|----------|------|
| **先修** | 触发失败、确认缺失、流程不可执行 | description 写错导致无法触发 |
| **再修** | 结构臃肿、重复内容、资源组织混乱 | 大量内容未拆分到 references |
| **最后** | 措辞润色、示例补充、可读性增强 | 措辞不够友好 |

---

## 适用场景

- 想检查和改进现有 skill 的质量
- skill 触发不准确、工作流混乱、输出不稳定
- skill 内容过于臃肿，需要拆分和重组
- 需要对 skill 进行结构化审查

## 不适用场景

- 从零创建一个新 skill（没有现有目标，用 prompt-craft）
- 只需要写一个 prompt（用 prompt-craft 或 prompt-optimize）

---

## 注意事项

| ❌ 不要这样做 | ✅ 正确做法 |
|--------------|-------------|
| 把"审查结论"和"直接开始改文件"混成一步 | 先输出计划，等待确认 |
| 为了显得全面就偷偷扩大改动面 | 围绕用户指定方向优化 |
| 在没确认前修改目标 skill | 等待用户明确确认 |
| 为了"审查完整"而把无关 reference 全部读进上下文 | 只读与本次优化相关的文件 |
