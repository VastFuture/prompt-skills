# 真实案例：提示词改造全过程

本节通过 3 个完整案例，展示如何组合使用多个技能解决实际问题。每个案例都包含问题诊断、改造过程和效果对比。

---

## 案例一：客服分流提示词

### 🎯 原始问题

提示词写了 200 多行，越写越乱，每次修改都要花 2 小时找位置。AI 响应不稳定，同样的问题偶尔答对偶尔答错。

### 🛠️ 使用的技能

- `prompt-refactor` — 系统诊断 + 重构
- `prompt-score` — 前后对比

### 📝 关键改造点

**重构前（问题片段）：**

```markdown
你是一个客服分流助手，用户会提出各种问题。

第一，如果用户问的是产品功能相关的问题，你需要...

第二，如果用户问的是价格相关的问题，你需要...

第三，如果用户问的是技术支持的问题，你需要...

还有一种情况是用户问的是物流问题，这时候你要...

[还有50多行类似规则...]
```

**问题诊断：**

| 问题类型 | 具体表现 |
|----------|----------|
| 规则碎片化 | 200 行分散成 50+ 条独立规则 |
| 漂移 | 不同分支对同一问题描述不一致 |
| 解释层过厚 | 「你需要」出现 30+ 次 |
| 边界泄漏 | 物流问题没有明确边界 |

**重构后（核心骨架）：**

```markdown
# 角色
客服分流专家，职责是根据问题类型转接到对应团队。

# 分流规则
| 问题类型 | 识别关键词 | 转接目标 |
|----------|-----------|----------|
| 产品功能 | 「怎么用」「功能」「设置」 | 产品组 |
| 价格咨询 | 「多少钱」「价格」「优惠」 | 销售组 |
| 技术支持 | 「报错」「不能用」「bug」 | 技术组 |
| 物流查询 | 「物流」「快递」「到了吗」 | 物流组 |

# 执行流程
1. 提取用户问题的核心关键词
2. 匹配分流规则表
3. 输出转接指令：{"team": "xxx", "message": "xxx"}
```

**重构五步法应用：**

```
Step 1: 通读全稿 → 发现问题：200行里有30多处「你需要」
Step 2: 标记问题 → 5类问题全部命中
Step 3: 收骨架 → 提取「分流」这一核心动作
Step 4: 改文案 → 表格替代碎片规则，「你需要」全部删除
Step 5: 验证 → 评分验证
```

### 📊 效果对比

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 提示词长度 | 200+ 行 | 45 行 |
| 规则数量 | 50+ 条碎片规则 | 1 张分流表 |
| 修改耗时 | 2 小时找位置 | 5 分钟改表格 |
| 响应一致性 | 偶尔答对 | 稳定答对 |
| 评分 | 3/10 | 8/10 |

### 💡 学到的经验

1. **规则碎片化是提示词腐烂的开始** — 一旦开始「再加一条」，就会无限增长
2. **表格比规则列表更稳定** — 结构化数据比自然语言更抗漂移
3. **核心动作只有一个** — 「分流」就是唯一目标，其他都是噪音

## 案例二：创意写作提示词

### 🎯 原始问题

AI 输出太模板腔，读起来像机器写的。用户反馈「感觉在读说明书」。

### 🛠️ 使用的技能

- `prompt-template` — 构建基础骨架
- `prompt-craft` — 金词优化 + 去除模板腔
- `prompt-principles` — 检查角色扮演问题

### 📝 关键改造点

**重构前：**

```markdown
你是一个创意写作助手，需要根据用户的主题创作故事。

要求：
1. 故事要有趣味性
2. 情节要生动
3. 人物要立体
4. 结尾要有意义
5. 文字要优美

请根据用户输入的主题创作一个完整的故事。
```

**问题诊断：**

| 问题 | 表现 |
|------|------|
| 约束堆砌 | 「要有」「要」出现 5 次 |
| 目标模糊 | 「趣味性」「生动」「优美」无法量化 |
| 无示例 | 没有告诉 AI 什么是「好的故事」 |
| 过度指令 | 5 条要求=0 条真正有效的约束 |

**六要素模板填充：**

```markdown
# Role
资深小说作者，风格独特，文字有温度。

# Background
用户需要针对给定主题创作故事，希望故事能打动人心。

# Task
创作一个 2000 字左右的短篇故事。

# Requirements
- 开篇 50 字内必须制造悬念或冲突
- 人物有明确缺陷或执念
- 结尾要反转或留白，不能直接说教
- 禁止出现「从此以后」「最后」「故事告诉我们」等模板句式

# Example
[附录：优秀短篇片段，分析悬念构建方式]

# Reflection
每次输出后检查：是否用了模板句式？人物是否有缺陷？
```

**金词替换：**

| 原词 | 替换 |
|------|------|
| 有趣味性 | 让人想一口气读完 |
| 情节生动 | 场景有画面感 |
| 人物立体 | 至少一个人物有读者看不透的一面 |
| 结尾有意义 | 读者会想很久 |

### 📊 效果对比

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 文字风格 | 「说明书」感 | 「文学」感 |
| 模板腔 | 「从此以后」「故事告诉我们」频繁出现 | 全部消除 |
| 读者反馈 | 「像机器写的」 | 「有温度」 |
| 评分 | 4/10 | 8/10 |

### 💡 学到的经验

1. **约束不是越多越好** — 5 条模糊约束 < 2 条清晰约束
2. **「要有」是提示词癌症** — 出现 3 次以上就要警惕
3. **示例比描述更有效** — 告诉 AI 什么是好，比告诉它怎么做更好

## 案例三：Coze 智能体

### 🎯 原始问题

在 Coze 上搭了一个客服机器人，变量对不上，JSON 输出格式乱。调试了 3 天还是有问题。

### 🛠️ 使用的技能

- `prompt-coze` — 修复变量定义和 JSON 对齐
- `prompt-score` — 最终验证

### 📝 关键改造点

**原始问题分析：**

```
用户反馈：
- 变量名和实际传入的数据对不上
- JSON 输出格式不稳定，有时多一个字段有时少一个
- AI 回答正确但格式不对，代码解析失败
```

**问题诊断：**

| 问题 | 根因 |
|------|------|
| 变量不对 | prompt 里写的变量名和 Coze 输入组件名不一致 |
| JSON 乱 | 没有明确 JSON 格式约束，AI 自由发挥 |
| 解析失败 | 输出内容里有多余的解释文字 |

**修复步骤：**

**Step 1: 对齐变量定义**

```markdown
# 输入变量（必须与 Coze 组件名称完全一致）
- user_message: 用户输入的文本
- user_level: 用户等级（"vip" | "normal"）
- order_id: 订单号（如无则为 "none"）

# 输出格式
严格 JSON，不要任何额外文字：
{"action": "转接"|"回答"|"结束", "target": "string", "reply": "string"}
```

**Step 2: 测试变量传递**

```
Coze 输入组件名: user_message
     ↓
Prompt 变量名: user_message ✓ 对齐

Coze 输入组件名: level
     ↓
Prompt 变量名: user_level ✗ 不对齐 → 修复为 level
```

**Step 3: 验证 JSON 输出**

```markdown
# 输出格式示例（严格遵循）
{"action": "回答", "target": "销售组", "reply": "您好，请问有什么可以帮您？"}

# 禁止出现的格式
- 带解释文字：「好的，以下是JSON输出：{...}」
- 换行格式：「{"action": "回答", ...}」
- 多余字段：增加不在示例里的字段
```

**完整修复后的 Prompt：**

```markdown
# 角色
客服分流助手，根据用户问题类型转接到对应团队。

# 输入变量
| Coze 组件名 | Prompt 变量名 | 说明 |
|------------|--------------|------|
| user_message | user_message | 用户输入 |
| level | user_level | 用户等级 |
| order_id | order_id | 订单号 |

# 分流规则
| 关键词 | action | target |
|--------|--------|--------|
| 价格/多少钱 | 转接 | 销售组 |
| 产品/功能 | 转接 | 产品组 |
| 技术/bug | 转接 | 技术组 |
| 其他 | 回答 | - |

# 输出格式
严格 JSON，无任何额外文字：
{"action": "转接"|"回答"|"结束", "target": "string", "reply": "string"}

# 示例
输入：这个东西多少钱？
输出：{"action": "转接", "target": "销售组", "reply": ""}
```

### 📊 效果对比

| 维度 | 修复前 | 修复后 |
|------|--------|--------|
| JSON 解析成功率 | 40% | 98% |
| 调试耗时 | 3 天 | 2 小时 |
| 格式稳定性 | 偶尔出错 | 稳定输出 |
| 评分 | 3/10 | 9/10 |

### 💡 学到的经验

1. **变量名必须与 Coze 组件名完全一致** — 差一个字母就不对
2. **JSON 输出必须严格约束** — 「不要额外文字」要写进去
3. **格式示例比描述更清晰** — 一个正确示例胜过 100 字说明

## 案例总结

### 共同模式

| 模式 | 出现场景 | 解决方案 |
|------|----------|----------|
| 规则碎片化 | 客服、复杂场景 | 表格化、结构化 |
| 模板腔 | 写作、创意场景 | 金词替换 + 示例 |
| 变量不对齐 | Coze 等平台 | 明确映射关系 |

### 评分变化

| 案例 | 改造前 | 改造后 | 提升 |
|------|--------|--------|------|
| 客服分流 | 3/10 | 8/10 | +5 |
| 创意写作 | 4/10 | 8/10 | +4 |
| Coze 智能体 | 3/10 | 9/10 | +6 |

### 核心原则

1. **先诊断再治疗** — 用 scorer 找到问题，再用对应技能修复
2. **结构优于列表** — 表格 > 规则，骨架 > 碎片
3. **示例胜于描述** — 一个正确示例顶 100 字说明
4. **约束要具体不要模糊** — 「让人想一口气读完」> 「有趣味性」
