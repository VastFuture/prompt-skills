# 实战工作流：从零到上线的提示词工程

本节介绍 5 个最常见场景的工作流，每个场景都配有清晰的流程图，帮助你快速选择正确的技能组合。

---

## 场景一：从零开始写提示词

这是最常见的场景。当你不确定从哪里开始时，推荐使用「模板 + 检查 + 评分」的组合流程。

**使用技能：** prompt-template → prompt-principles → prompt-score

### 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                    从零开始写提示词                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Step 1: 确定骨架  │
                    │ efficient-      │
                    │ prompt-template  │
                    │ -gemini          │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 2: 检查要素  │
                    │ prompt-     │
                    │ best-practices  │
                    │ 确认三要素齐全    │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 3: 评分验证  │
                    │ prompt-       │
                    │ scorer        │
                    │ 1-10 分打分     │
                    └────────┬─────────┘
                             │
              ┌──────────────┴──────────────┐
              │                                 │
              ▼                                 ▼
         分数 ≥ 8                           分数 < 8
         完成                               进入优化流程
              │                                 │
              ▼                                 ▼
         结束                           回到 Step 2 修改
```

### 详细步骤

| 步骤 | 操作 | 目标 |
|------|------|------|
| 1 | 用六要素模板（Role / Background / Task / Requirements / Example / Reflection）填充骨架 | 快速获得基础版本 |
| 2 | 对照三要素检查（Goal Clarity / Constraint Flow / Workflow） | 确保要素完整 |
| 3 | 用评分器打分，低于 8 分则返回第 2 步修改 | 保证质量底线 |

## 场景二：提示词效果不好，想优化

当你已经有提示词但效果不佳时，先诊断再治疗。

**使用技能：** prompt-score → prompt-principles → prompt-optimize

### 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                  提示词效果不好，想优化                         │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Step 1: 诊断问题  │
                    │ prompt-       │
                    │ scorer        │
                    │ 找出具体问题点   │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 2: 定位根因  │
                    │ prompt-      │
                    │ principles   │
                    │ -engineer       │
                    │ 找到问题本质    │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 3: 匹配框架  │
                    │ prompt-optimize │
                    │ 57种框架选最优  │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 4: 修改验证  │
                    │ 修改后重新打分   │
                    │ 确认达到 8+     │
                    └─────────────────┘
```

### 常见问题与对应框架

| 问题类型 | 推荐框架 |
|----------|----------|
| 角色扮演太多导致废话 | 精简版 CRISPE |
| 指令太模糊 | RACEF / FOCUS |
| 输出格式不稳定 | TRACE / ORID |
| 推理能力不足 | Chain of Thought |
| 创意不足 | SCAMPER / SPARK |

## 场景三：提示词太长太乱

当提示词超过 200 行，逻辑混乱时，需要系统性重构。

**使用技能：** prompt-refactor → prompt-score

### 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                    提示词太长太乱                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Step 1: 诊断问题  │
                    │ prompt-refactor │
                    │ 系统诊断5类问题   │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 2: 重构五步  │
                    │ 通读全稿→标记问题 │
                    │ →收骨架→改文案   │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 3: 验证质量  │
                    │ prompt-       │
                    │ scorer        │
                    └────────┬─────────┘
                             │
              ┌──────────────┴──────────────┐
              │                                 │
              ▼                                 ▼
         分数 ≥ 8                           分数 < 8
         完成                               返回 Step 2
              │                                 │
              └──────────────┬─────────────────┘
                             │
                             ▼
                        结束
```

### 重构五步法详解

1. **通读全稿** — 不做任何修改，先完整读一遍
2. **标记问题** — 用 Template Smell 清单找出问题点
3. **收骨架** — 提取核心流程，删除冗余解释
4. **改文案** — 按 5 条改写铁律重写
5. **验证** — 用评分器确认质量

## 场景四：在 Coze 写智能体

Coze 平台有其特殊性：变量定义、JSON 输出、触发词设计都有讲究。

**使用技能：** prompt-coze → prompt-score

### 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                    在 Coze 写智能体                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Step 1: 写基础版  │
                    │ coze-prompt-   │
                    │ writing        │
                    │ 变量+JSON格式   │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 2: 结构优化  │
                    │ prompt-       │
                    │ scorer        │
                    │ 提升完整度评分   │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 3: 测试验证  │
                    │ 测试变量传递     │
                    │ 确认JSON输出正确 │
                    └─────────────────┘
```

### Coze 特有检查项

- 输入变量是否正确定义
- 输出 JSON 格式是否与代码预期一致
- 触发词是否清晰、不冲突
- 工作流节点是否逻辑连贯

## 场景五：开发 Claude Code Skill

Skill 开发比普通提示词更复杂，需要考虑触发机制、工作流设计和资源组织。

**使用技能：** skill-optimize → prompt-craft → prompt-refactor → prompt-score

### 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                    开发 Claude Code Skill                    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ Step 1: 审查设计  │
                    │ skill-optimize │
                    │ 触发/工作流/资源  │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 2: 构建核心  │
                    │ prompt-craft    │
                    │ 7维修炼+骨架    │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 3: 优化结构  │
                    │ prompt-refactor │
                    │ 结构化重组      │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │ Step 4: 最终检查  │
                    │ prompt-       │
                    │ scorer        │
                    │ 确认达到 8+     │
                    └─────────────────┘
```

### Skill 审查重点

| 维度 | 检查项 |
|------|--------|
| 触发 | description 是否同时说明「做什么」和「何时用」 |
| 模式 | Tool Wrapper / Generator / Reviewer 是否匹配 |
| 工作流 | 是否清晰可执行 |
| 资源 | references/scripts/assets 组织是否合理 |
| 输出 | 是否方便另一个 agent 稳定执行 |

