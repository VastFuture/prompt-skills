# 提示词原则：核心法则 + 工程师实践

> **核心理念**：对 AI 诚实、说清楚、不折腾。像和能干的新同事说话那样写提示词。

---

## 一句话描述

Universal 原则（目标清晰 + 约束流）+ Anthropic 工程师一线实践（直接沟通 + 迭代 + 诚实），两个视角合一。

---

## Part 1：通用原则（Universal）

### 原则一：一切字符可描述的问题皆可解

> "Anything that can be described in text can be solved by an LLM."

只要你能用文字描述清楚你的问题，AI 就有可能帮你解决。

**这个原则告诉我们**：
- 不要害怕问题复杂，只要你能描述清楚
- 提示词的本质是**描述能力**的竞赛
- 写不好提示词，往往是想不清楚自己要什么

**实践中的含义**：
- 遇到 AI 输出不理想，先问自己：我的需求描述清楚了吗？
- 与其说"帮我优化"，不如说"把这段话从200字压缩到50字，保留核心观点"
- AI 的能力边界取决于你描述问题的精确度

### 原则二：目标清晰 + 约束流

写提示词最关键的三件事：

#### 1. Goal（目标）— 你要去哪？

明确、具体、可达成。

```
❌ 模糊目标：帮我写点东西
✅ 清晰目标：写一段50字的产品介绍，目标用户是大学生，强调性价比

❌ 模糊目标：分析一下数据
✅ 清晰目标：分析这批用户评论，识别出主要的抱怨点和赞扬点，各输出3条
```

#### 2. Constraints（约束）— 你要怎么去？

定义输出的形式、风格、边界。

```
✅ 有约束：输出为表格，两列分别是"问题类型"和"频率"
✅ 有约束：用中文回答，避免专业术语
✅ 有约束：总字数不超过300字
```

#### 3. Workflow（工作流）— 分几步走？

把复杂任务拆成可执行的步骤。

```
✅ 有工作流：
1. 先阅读用户评论，识别情绪
2. 分类：抱怨/赞扬/建议/其他
3. 统计各类频率
4. 输出汇总表格
```

## Part 2：Anthropic 工程师最佳实践

### 原则一：直接沟通，不绕弯子

**核心理念**：把 AI 当成一个能干的远程同事，直接说你想要什么。

| ❌ 绕弯子的写法 | ✅ 直接的写法 |
|---------------|-------------|
| "你现在是一个资深产品经理，你需要..." | "写一段产品介绍" |
| "我想请你帮我分析一下这个问题，如果你有时间的话" | "分析这份用户反馈，找出3个核心问题" |
| "作为一个AI语言模型，我需要你..." | "把这段话翻译成英文" |

**为什么直接更好？**
- 节省 token
- 减少理解歧义
- AI 更容易理解你的真实意图

### 原则二：迭代式优化，不要一步到位

**核心理念**：提示词工程是一个试错过程，不是一次性设计。

**正确心态**：
- 不期望第一次就写出完美的提示词
- 每次运行后问："为什么这次结果不理想？是我的提示词哪里没说清楚？"
- 边测边改，逐步逼近目标

**测试边界情况**：
- 空输入时 AI 会怎样？
- 数据异常时 AI 会怎样？
- 边界条件（如超长文本、特殊字符）时 AI 会怎样？

**迭代模板**：
```
第一次：AI输出 → 分析哪里不对
第二次：调整提示词 → 测试
第三次：继续微调 → 测试
...直到满意
```

### 原则三：对模型诚实，不欺骗

**核心理念**：不要假装自己是老师出题，不要骗 AI "这是一个测试"。

| ❌ 欺骗性写法 | ✅ 诚实写法 |
|-------------|-----------|
| "我在测试你的能力，请认真回答" | 直接说明你的真实需求 |
| "这是一个学术研究项目" | "我需要你帮我整理一份竞品分析报告" |
| "回答错误会扣分" | "这个任务需要准确的数据" |

**为什么诚实更好？**
- 现代 AI 已经足够强大，不需要"哄骗"
- 欺骗性提示词会让 AI 产生防御心态
- 诚实描述让 AI 专注于你的真实目标

### 原则四：不要过度角色扮演

**核心理念**：AI 不需要你假装它是谁，它需要的是清楚的任务描述。

| ❌ 过度角色扮演 | ✅ 简单直接 |
|--------------|-----------|
| "你是一个有20年经验的法律顾问，毕业于哈佛大学..." | "帮我分析这份合同的风险点" |
| "你是一位温柔耐心的老师，用循循善诱的方式..." | "用简单的语言解释量子计算" |
| 长篇大论的假背景设定 | 直接说你的需求 |

**什么时候需要角色？**
- 角色能帮助 AI 理解回答风格时（如"用幽默的方式回答"）
- 角色能提供专业背景知识时

**什么时候不需要？**
- 简单任务不需要角色
- 过度角色扮演反而会增加 AI 的"幻觉"风险

### 原则五：信任模型的理解能力

**核心理念**：把论文直接给 AI，不要擅自简化成"小白版本"。

| ❌ 低估模型 | ✅ 信任模型 |
|-----------|-----------|
| 把专业论文改写成"科普版"再给AI | 直接给论文，让AI自己提炼 |
| 担心AI"看不懂"，过度解释背景 | 信任AI的上下文理解能力 |
| 把复杂问题拆成100个简单问题 | 相信AI能处理复杂任务 |

**什么该信，什么不该信**：
- AI 能理解复杂指令 → 信
- AI 的事实准确性（有幻觉风险）→ 需核实
- AI 的数学计算 → 需复核
- AI 的专业领域知识 → 视情况核实

## 常见错误 Top 5

### 错误一：目标模糊

```
❌ 帮我分析一下这个产品
❌ 看看这段代码有什么问题
❌ 优化一下这个文案
```

**问题**：AI 不知道你要分析什么维度、输出什么格式、详细到什么程度。

**改进**：
```
✅ 分析这个产品的3个最大增长机会，说明每个机会的实现难度和预期收益
✅ 找出这段代码中影响性能的3个问题，每个问题给出具体改进方案
✅ 把这段文案压缩到100字以内，保留核心卖点，去掉重复信息
```

### 错误二：约束缺失

```
❌ 写一篇关于AI的文章
✅ 写一篇800字以内的科普文章，面向零基础读者，语言通俗易懂，包含3个生活中的例子
```

**问题**：没有约束，AI 按自己的理解输出，结果可能完全不是你想要的。

### 错误三：期望 AI "猜测"你的意思

```
❌ 用户："帮我写个需求文档"
  AI：输出了一份标准PRD
  用户："不是这个格式"
  （来回修改5次）

✅ 用户："帮我写一个面向开发团队的功能需求文档，包含：功能描述、输入输出、边界情况、错误处理"
```

**问题**：期望 AI "理解你的意思"而不是"执行你的指令"。

### 错误四：过度设计提示词

```
❌ 写了一个500字的复杂提示词，希望"一步到位"
✅ 先写简单的，看结果再迭代
```

**事实**：花 2 小时写一个完美提示词，不如花 10 分钟写初稿、测试、迭代。提示词工程是实验科学，不是理论设计。

### 错误五：不敢直接批评 AI 输出

```
❌ "也许AI是对的，我的要求可能有问题"
✅ "这个结果不符合预期，告诉我哪里出问题了"
```

**事实**：当 AI 输出不理想时，第一反应应该是检查提示词，而不是替 AI 找借口。

## 最佳实践 Top 3

### 实践一：录下你的需求，转录给 AI

**技巧**：想象你在和同事视频会议，直接说出你的需求，录音转录后粘贴给 AI。

**效果**：通常比自己"优化"过的提示词效果更好，因为保留了真实的思考过程。

### 实践二：明确输出格式

**技巧**：告诉 AI 你期望的输出格式，而不是让它"自己决定"。

```
❌ "分析这段文本"
✅ "分析这段文本，输出格式：{主题: string, 情感: positive/negative/neutral, 关键词: string[]}"
```

### 实践三：测试后再正式使用

**技巧**：在上线前，用边界条件测试你的提示词。

- 空输入
- 超长输入
- 异常格式
- 极端情况

## 快速检查清单

发送提示词之前，快速自检：

- [ ] **目标清晰吗？** — 能用一句话说清楚你要什么吗？
- [ ] **约束够吗？** — 至少写了 3 条输出约束？
- [ ] **工作流明确吗？** — 需要分步骤吗？步骤写清楚了吗？
- [ ] 我的提示词是直接说需求，还是在绕弯子？
- [ ] 我对 AI 诚实、不欺骗吗？
- [ ] 我测试过边界情况吗？
- [ ] 我是否过度角色扮演了？
