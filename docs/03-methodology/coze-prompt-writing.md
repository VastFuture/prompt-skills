# Coze 平台提示词写法

> **核心理念**：Coze 的提示词本质是 Agent 的"说明书"，核心是变量格式对齐——Agent 的输出格式必须与输入/输出变量的格式完全一致。

---

## 一句话描述

Coze 平台上写提示词的关键规则：JSON 格式变量 + Agent 描述与输出变量一一对应 + 输出格式与变量格式对齐 + 数据库 Schema 匹配。

---

## Coze 提示词 vs 普通提示词

| 对比维度 | 普通提示词 | Coze 提示词 |
|---------|-----------|-----------|
| **核心关注点** | 任务完成质量 | 变量格式对齐 |
| **输出格式** | 任意格式 | 必须符合预定义的变量格式 |
| **Agent 描述** | 可灵活书写 | 必须与输出变量一一对应 |
| **数据库集成** | 无需考虑 | 需要匹配 DB Schema |
| **调试方式** | 直接测试 | 需要验证变量传递链路 |
| **复用性** | 一次性使用 | 可作为工作流节点复用 |

---

## 四大关键规则

### 规则一：变量输出必须使用 JSON 格式

**核心要求**：Coze 中所有涉及变量操作的输出，必须使用 JSON 格式。

```
❌ 错误写法：
"输出结果为：姓名-张三，年龄-30岁"

✅ 正确写法：
"输出结果为JSON格式：
{
  "name": "张三",
  "age": 30
}"
```

**原因**：Coze 的变量传递基于 JSON，只有 JSON 格式才能被后续节点正确读取和处理。

---

### 规则二：Agent 描述与输出变量一一对应

**核心要求**：Agent 描述中提到的每个输出，必须有对应的输出变量定义。

```
Agent 描述示例：
"根据用户输入的城市名称，查询该城市的天气情况，
返回：城市名、天气、温度、湿度、空气质量"

对应的输出变量定义：
- city (string): 城市名
- weather (string): 天气情况
- temperature (number): 温度
- humidity (number): 湿度
- aqi (number): 空气质量指数
```

**对应关系检查**：
| Agent 描述中提到的输出 | 是否定义输出变量 |
|----------------------|----------------|
| 城市名 | ✅ city (string) |
| 天气 | ✅ weather (string) |
| 温度 | ✅ temperature (number) |
| 湿度 | ✅ humidity (number) |
| 空气质量 | ✅ aqi (number) |

---

### 规则三：Agent 输出格式必须与输出变量格式对齐

**核心要求**：提示词中描述的输出格式，必须与输出变量的实际格式完全一致。

| 输出变量定义 | Agent 描述应体现的格式 |
|-------------|---------------------|
| `name: string` | 输出应为字符串，如 "张三" |
| `age: number` | 输出应为数字，如 30 |
| `tags: string[]` | 输出应为数组，如 ["标签1", "标签2"] |
| `metadata: object` | 输出应为对象，如 `{key: value}` |

---

### 规则四：数据库 Schema 匹配

**核心要求**：如果数据需要存入数据库，输出变量格式必须与 DB Schema 匹配。

```
数据库 Schema：
CREATE TABLE products (
  id: INTEGER PRIMARY KEY,
  name: TEXT NOT NULL,
  price: REAL,
  category: TEXT,
  tags: TEXT  -- JSON字符串格式存储
);

对应的输出变量：
- id: number
- name: string
- price: number
- category: string
- tags: string (JSON字符串)
```

## 变量定义指南

### 输入变量定义

在"用户提示词模块"中定义：

```markdown
## 输入变量说明
- {{variable_name}}: [变量说明], 格式: [类型], 示例: [示例值]
```

### 输出变量定义

在 Agent 描述中明确，并确保与输出变量配置一致：

```markdown
## 输出变量说明
- {{output_variable_name}}: [变量说明], 格式: [类型], 示例: [示例值]
```

## JSON 格式示例

### 示例一：简单对象

**输出变量定义**：
```json
{
  "name": "产品名称",
  "type": "string",
  "price": 99.9,
  "type": "number"
}
```

**Agent 描述示例**：
```
输出结果为JSON格式：
{
  "name": "产品名称（如：无线蓝牙耳机）",
  "price": 99.9
}
```

---

### 示例二：数组对象

**输出变量定义**：
```json
{
  "products": [
    {
      "id": 1,
      "name": "产品名",
      "price": 99.9
    }
  ]
}
```

**Agent 描述示例**：
```
输出结果为JSON数组格式：
[
  {
    "id": 1,
    "name": "产品名称",
    "price": 99.9
  },
  {
    "id": 2,
    "name": "产品名称",
    "price": 199.9
  }
]
```

---

### 示例三：嵌套对象

**输出变量定义**：
```json
{
  "user": {
    "name": "用户名",
    "type": "string"
  },
  "orders": ["订单列表", "array"]
}
```

**Agent 描述示例**：
```
输出结果为JSON格式：
{
  "user": {
    "name": "张三"
  },
  "orders": [
    {"order_id": "A001", "amount": 100},
    {"order_id": "A002", "amount": 200}
  ]
}
```

## 常见错误 Top 3

### 错误一：输出格式与变量类型不匹配

```
❌ 定义：age: number
   输出："三十岁"（字符串）

✅ 定义：age: number
   输出：30（数字）
```

**后果**：后续节点无法正确读取和处理。

---

### 错误二：Agent 描述的输出多于或少于变量定义

```
❌ Agent 描述提到 5 个输出，但只定义了 3 个输出变量
   → 后续节点只能用到 3 个

✅ Agent 描述的输出数量 = 输出变量数量
   → 完全对应
```

**检查方法**：数一数 Agent 描述中承诺的输出项，是否与输出变量配置一一对应。

---

### 错误三：JSON 格式不合法

```
❌ "输出格式：name-张三，age-30"
   （不是合法JSON）

✅ 输出格式：
{
  "name": "张三",
  "age": 30
}
```

**后果**：后续节点读取失败，工作流中断。

## 适用场景

| 场景 | 推荐度 | 说明 |
|------|--------|------|
| Coze 工作流搭建 | ⭐⭐⭐⭐⭐ | 核心使用场景 |
| 多 Agent 协作 | ⭐⭐⭐⭐⭐ | 变量传递必须对齐 |
| 数据库集成 | ⭐⭐⭐⭐⭐ | Schema 匹配是关键 |
| API 接口开发 | ⭐⭐⭐⭐ | 输出需要 JSON 格式 |
| 简单对话机器人 | ⭐⭐⭐ | 简单场景可用默认格式 |

---

## 不适用场景

| 场景 | 不推荐原因 |
|------|-----------|
| 纯对话交互 | 不需要变量传递 |
| 一次性问答 | 不需要复用 |
| 外部 API 调用 | 可能需要其他格式 |

---

## 快速检查

- [ ] 变量操作输出是否使用 JSON 格式？
- [ ] Agent 描述中的输出项是否都有对应变量？
- [ ] 输出格式是否与变量类型一致？（string/number/array）
- [ ] JSON 格式是否合法？（可用 JSONLint 验证）
- [ ] 有数据库集成时，格式是否匹配 DB Schema？

## 一图总结

```
Coze 提示词开发流程：

  ┌─────────────────────────────────────────────┐
  │ 1. 设计输出变量                              │
  │    定义变量名、类型、格式                     │
  └──────────────────┬──────────────────────────┘
                     ▼
  ┌─────────────────────────────────────────────┐
  │ 2. Agent 描述                                │
  │    按变量定义格式描述输出                     │
  │    一一对应，不多不少                        │
  └──────────────────┬──────────────────────────┘
                     ▼
  ┌─────────────────────────────────────────────┐
  │ 3. 测试验证                                  │
  │    验证 JSON 格式合法                        │
  │    验证变量传递正确                          │
  └──────────────────┬──────────────────────────┘
                     ▼
  ┌─────────────────────────────────────────────┐
  │ 4. 连接数据库（如需要）                       │
  │    确保 Schema 匹配                          │
  └─────────────────────────────────────────────┘
```

> **提示**：Coze 提示词开发的核心是"格式对齐"。把这一点做好，后面的工作流搭建和调试会顺利很多。
