# 提示词评分与质检

> **核心理念**：好的提示词不是靠感觉，而是可以量化评估的。同时，推理模型（O1/O3-mini）有特殊要求，需要专属检查逻辑。

---

## 一句话描述

通用提示词 10 维评分（1-10分）+ 推理模型专属质检（O1/O3-mini），自动识别类型，走对应分支。

---

## 如何选择分支

```
你的提示词是给谁的？
│
├── O1 / O3-mini / 推理模型 ──→ 推理模型质检分支
│
└── 普通模型（GPT-4/Claude/Gemini等）──→ 通用评分分支
```

---

# 分支一：通用评分（普通模型提示词）

> **核心理念**：10个维度，找出问题所在。分数不是目的，改进才是。

## 评分维度（10项，每项1-10分）

| # | 维度 | 评估问题 | 10分标准 |
|---|------|---------|---------|
| 1 | **Clarity** | 意图是否清晰？指导信息是否充分？ | 目标明确，无歧义 |
| 2 | **Relevance** | 内容是否紧扣任务？引导相关回复？ | 完全相关，不跑题 |
| 3 | **Completeness** | 必要元素是否完整？ | 所有必要元素齐全 |
| 4 | **Neutrality** | 是否避免了引导性语言或偏见？ | 完全中立 |
| 5 | **Creativity** | 是否鼓励创新性思维？ | 充分鼓励创新 |
| 6 | **Structure** | 是否有助于预期响应路径？ | 结构清晰引导性强 |
| 7 | **Grammar** | 语法是否正确？ | 完全正确 |
| 8 | **Fluency** | 语言是否自然流畅？ | 非常流畅 |
| 9 | **Goal Alignment** | 是否与原始意图一致？ | 完全一致 |
| 10 | **Testability** | 是否可靠、一致地测试？ | 完全可测试 |

## 分数含义

| 分数区间 | 评价 | 说明 |
|---------|------|------|
| **1-3分** | 需重建 | 结构性问题严重，需要重新设计 |
| **4-6分** | 及格/需改进 | 有基本框架，但有明显缺陷 |
| **7-9分** | 良好/优秀 | 质量较高，可投入使用 |
| **10分** | 完美 | 理论最优，实际少见 |

综合评分 = 10项平均分：

| 综合评分 | 评价 | 建议 |
|---------|------|------|
| **8.5-10** | 优秀 | 可直接使用 |
| **7.0-8.4** | 良好 | 微调后使用 |
| **5.0-6.9** | 及格 | 需要较多改进 |
| **<5.0** | 较差 | 建议重新设计 |

## 评分输出格式

```
【综合评分】X.X / 10

【分项评分】
- Clarity: X/10 | Relevance: X/10 | Completeness: X/10
- Neutrality: X/10 | Creativity: X/10 | Structure: X/10
- Grammar: X/10 | Fluency: X/10 | Goal Alignment: X/10 | Testability: X/10

【最低分项分析】
1. [维度名称] (X/10) — 原因分析
2. [维度名称] (X/10) — 原因分析
3. [维度名称] (X/10) — 原因分析

【改进建议】(3条)
1. [建议内容] — [原文引用] — [改进机制说明]
2. [建议内容] — [原文引用] — [改进机制说明]
3. [建议内容] — [原文引用] — [改进机制说明]

【改进后提示词】
[完整改进版本，加粗标注修改处]
```

## 使用示例

**提交**：`帮我写一封商务邮件，催促进度。`

**评分结果**：
```
【综合评分】4.2 / 10

【分项评分】
- Clarity: 3/10  ← 目标模糊
- Relevance: 5/10
- Completeness: 3/10  ← 缺少约束
- Neutrality: 7/10
- Creativity: 4/10
- Structure: 3/10  ← 无格式要求
- Grammar: 9/10
- Fluency: 7/10
- Goal Alignment: 4/10  ← 目标不明确
- Testability: 3/10  ← 无法验证完成度

【改进建议】
1. 将"写邮件"改为"写一封面向项目经理的正式商务邮件，200字以内" —
   [原文]"帮我写一封商务邮件" —
   [机制]具体化目标让AI明确输出边界，减少无效迭代

2. 增加格式约束"包含称呼、正文、落款三部分，使用敬语" —
   [原文]无格式要求 —
   [机制]明确定义结构让输出更可预测

3. 增加语气约束"语气主动但不催促，表达合作诚意" —
   [原文]无语气要求 —
   [机制]避免AI按默认风格输出可能不合适的语气

【改进后提示词】
写一封面向**项目经理**的**正式商务邮件**，**200字以内**，催促进度。
语气要求：**主动但不催促，表达合作诚意**。
格式要求：包含**称呼、正文、落款**三部分，使用敬语。
```

# 分支二：推理模型质检（O1/O3-mini）

> **核心理念**：推理模型有内置的深度思考能力，提示词要给它空间去思考，而不是替它思考。

## 普通模型 vs 推理模型

| 对比维度 | 普通模型（如GPT-4） | 推理模型（如O1/O3-mini） |
|---------|-------------------|----------------------|
| **工作方式** | 接收输入，直接输出 | 接收输入，**内部思考后**输出 |
| **思考能力** | 需要在提示词中引导 | 内置深度思考能力 |
| **链式推理** | "let's think step by step" | 模型自动执行 |
| **适用任务** | 通用对话、快速问答 | 复杂推理、数学、代码 |

**类比**：
```
普通模型 = 计算器
  → 你告诉它按什么公式计算

推理模型 = 数学家
  → 你给它一道数学题，它自己推理
  → 你不需要告诉它"用微积分"
```

## 推理模型的关键规则

### 规则一：禁用链式推理提示

```
❌ 禁用："let's think step by step"
❌ 禁用："请逐步思考"
❌ 禁用："先分析问题，再给出结论"
```

**原因**：推理模型已经有内置的思考机制，这些提示词会干扰它的正常推理。

### 规则二：给目标，不给过程

```
❌ 普通模型写法："先阅读文本，再提取关键词，最后分类"
✅ 推理模型写法："分析这段文本的情感倾向和主要观点"

❌ 普通模型写法："请按以下5个步骤操作..."
✅ 推理模型写法："作为客服AI，根据用户描述的问题，提供解决方案"
```

### 规则三：目标和约束要更精确

- **目标要精确**：让它知道你要什么结果
- **约束要清晰**：让它知道边界在哪里
- **上下文要充足**：它需要足够信息来推理

## 模型选择建议

| 任务复杂度 | 步骤数 | 推荐模型 | 提示词风格 |
|-----------|--------|---------|-----------|
| **简单** | 1-3步 | O3-mini | 简洁、目标明确 |
| **中等** | 3-5步 | O3-mini | 增加上下文和约束 |
| **复杂** | 5步以上 | O1 | 完整背景 + 充分约束 |
| **超大数据量** | 128k+ tokens | O3-mini | 需要预处理数据 |
| **超大数据量** | >200k tokens | 预处理后使用 | 不能直接用 |

## 参数建议

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **reasoning_effort** | low/medium/high | 根据任务复杂度选择 |
| **max_completion_tokens** | 500-25000 | 简单任务500，复杂任务25000 |
| **temperature** | ≤0.3 | 推理模型需要稳定性 |

## 常见错误 Top 3

### 错误一：使用链式推理提示

```
❌ "let's think step by step" — 会干扰推理模型的内部机制！
```

### 错误二：过度约束推理空间

```
❌ "必须按照以下5个步骤分析..." — 推理模型需要空间去思考最优路径！
```

### 错误三：混淆模型类型

```
❌ 简单翻译任务用 O1
❌ 复杂推理任务用 GPT-4
```

## 适用场景

| 场景 | 推荐度 | 说明 |
|------|--------|------|
| 数学/逻辑推理 | ⭐⭐⭐⭐⭐ | O1 强项 |
| 代码调试/优化 | ⭐⭐⭐⭐⭐ | 推理能力关键 |
| 复杂数据分析 | ⭐⭐⭐⭐ | 需要充分上下文 |
| 战略规划/决策分析 | ⭐⭐⭐⭐ | 多因素推理 |
| 快速问答/简单翻译 | ⭐⭐ | 用普通模型更高效 |
| 创意写作/内容生成 | ⭐⭐ | 推理模型不是最佳选择 |

# 通用检查清单

发送提示词之前，快速自检：

- [ ] **目标清晰吗？** — 能用一句话说清楚你要什么吗？
- [ ] **约束够吗？** — 至少写了 3 条输出约束？
- [ ] **工作流明确吗？** — 需要分步骤吗？步骤写清楚了吗？
- [ ] **这是给推理模型的吗？** — 如果是，确认没有链式推理提示
- [ ] **模型选对了吗？** — 简单任务用 O3-mini，复杂推理用 O1

提示词质检流程：

  ┌──────────┐
  │ 提交提示词 │
  └────┬─────┘
       │
       ▼
  ┌─────────────────┐
  │ 是推理模型提示词？ │ ──否──→ 通用评分（10维度）
  └──┬──────────┬──┘
     是          │
